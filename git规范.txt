1. 分支模型（Branch Model）

主分支： • main/master —— 永远保持可发布

开发分支： 
• develop —— 常规开发 
• feature/* —— 新功能 
• fix/* —— Bug修复 
• release/* —— 预发布 
• hotfix/* —— 紧急修复线上

分支命名示例： feature/user-auth fix/navbar-style release/v2.1.0

------------------------------------------------------------------------

2. Commit 提交规范（Angular）

格式： ():

常见 Type： 
• feat —— 新功能 
• fix —— 修复问题 
• docs —— 文档更新 
• style —— 格式调整（非逻辑） 
• refactor —— 重构 
• perf —— 性能优化 
• test—— 测试内容 
• chore —— 构建脚本、配置 
• revert —— 回滚

示例： feat(login): 添加登录功能 fix(api): 修复用户数据加载失败

------------------------------------------------------------------------

3. Pull Request 规范

PR 命名： [feature] 完成用户登录功能 [fix] 修复订单接口错误

PR 要求： 
• 描述清晰 
• UI 变更附截图 
• 关联 issue 
• 单一职责，不要一次提交过多内容

------------------------------------------------------------------------

4. 代码 Review 规范

检查内容： 
• 命名规范 
• 无重复代码 
• 无潜在 bug 
• 性能是否受影响 
• 符合项目架构 
• 是否包含必要测试
• 未确认前不合并

------------------------------------------------------------------------

5. 版本号规范（SemVer）

语义化版本号： MAJOR.MINOR.PATCH

示例： 1.0.0 —— 初始版本 
      1.1.0 —— 新功能 
      1.1.1 —— Bug 修复 
      2.0.0 ——大版本变更

------------------------------------------------------------------------

6. 提交前检查事项（Checklist）

✔ 代码可运行 
✔ 测试通过 
✔ 无 console.log 等调试信息 
✔ 文档已更新 
✔ Commit 信息规范 
✔ PR 内容完整

------------------------------------------------------------------------

7. 最佳实践（Best Practices）

• 每次提交只做一件事 
• 定期删除已合并分支 
• 使用 .gitignore 
• 使用rebase 保持提交历史整洁 
• 不合并未 review 的代码
